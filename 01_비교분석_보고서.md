# [프로젝트 1] 자동화 도구 비교 구현 — Make vs Zapier

## 1. 공통 워크플로우 정의

> **"피드백 수집 폼 접수 → 만족도 분기 → Discord 채널별 알림 + Google Sheets 기록"**

| 단계 | 구성 요소 |
|------|-----------|
| Trigger | Google Form 신규 응답 감지 |
| 조건 분기 | 만족도 = 만족 / 불만족 |
| Action 1 | [분기별] 해당 Discord 채널에 피드백 알림 전송 |
| Action 2 | Google Sheets 결과 행 기록 |

두 도구 모두 **동일한 구조**로 구현하여, 같은 입력에 같은 결과가 나오도록 맞춤.

---

## 2. 도구별 구현 과정 요약


### 2-1. Make
- 시나리오 캔버스에 모듈을 노드로 배치
- `Google Sheets Watch New Rows → Router(2분기) → Discord / Google Sheets`
- Router 모듈로 두 갈래를 만들고 각 경로에 필터 조건 설정
  - 1st 경로: `만족도 = 만족` → Discord `#긍정피드백` 채널
  - 2nd 경로: `만족도 = 불만족` → Discord `#개선요청` 채널
- `Run once`로 수동 트리거 후, 각 모듈 데이터 버블로 결과 확인

### 2-2. Zapier
- 에디터에서 단계를 위→아래 리스트로 추가
- `Google Forms New Response → Paths(2분기) → Discord / Google Sheets`
- Paths 모듈로 두 갈래를 만들고 각 Path에 조건 설정
  - Path A: `만족도 = 만족` → Discord `#긍정피드백`
  - Path B: `만족도 = 불만족` → Discord `#개선요청`
- 각 단계마다 `Test step`으로 검증, Zap History에서 실행 로그 확인

## 4. 비교 분석 (8개 항목)

| # | 비교 항목 | Make | Zapier |
|---|-----------|------|--------|
| 1 | UI/UX | 시각적 노드(캔버스) 기반. 분기·흐름이 한눈에 보임 | 위→아래 리스트(스텝) 기반. 선형 흐름에 직관적 |
| 2 | 설정 난이도 | 초기 학습 곡선 있음(매핑·라우터 개념 이해 필요) | 진입장벽 낮음, 클릭 위주로 빠르게 구성 가능 |
| 3 | 조건 분기 | Router/Filter 기본 제공, 무료에서도 사용 가능 | Paths는 유료(Pro) 전용, 무료는 단순 2스텝만 |
| 4 | 무료 플랜 범위 | 1,000 크레딧/월, 활성 시나리오 2개, 15분 간격 | 100 tasks/월, 2스텝 제한, 멀티스텝·분기 불가 |
| 5 | 연동 서비스 범위 | 3,000+ 앱, HTTP/Webhook 모듈로 자유 확장 | 8,000+ 앱, 커넥터 수 자체는 더 많음 |
| 6 | 실행 로그 확인 | 실행 히스토리에서 모듈별 입·출력 데이터까지 확인 | Zap History에서 task 단위 성공/실패 확인 |
| 7 | 과금 모델 | 모듈 실행당 1 크레딧 소모 | 액션당 1 task 소모, 동일 워크플로우 기준 비용 높음 |
| 8 | 에러 처리 | Error handler 라우트로 대체 경로 설계 유연 | Autoreplay(자동 재시도)는 유료 플랜에서만 가능 |

> 무료 플랜 수치는 2026년 6월 기준이며, 정책 변경 시 공식 페이지에서 재확인 필요.

## 5. 각 도구의 장단점

### Make
- **장점**: 무료에서도 분기·멀티스텝 모두 가능. 시각적 캔버스로 흐름 파악 쉬움. 모듈별 데이터가 투명하게 보여 디버깅 편리. 에러 처리 설계가 유연함.
- **단점**: 처음 사용 시 라우터·매핑 개념 학습 필요. 스텝이 많을수록 크레딧 소모 빠름. 무료는 15분 폴링 간격 제한.

### Zapier
- **장점**: 가장 직관적인 UX. 커넥터 수가 압도적으로 많음. 학습 자료 풍부. 단순 2앱 연결은 가장 빠르게 완성 가능.
- **단점**: 무료 플랜에서 분기·멀티스텝 불가. Paths·재시도 등 핵심 기능이 전부 유료. 동일 워크플로우 기준 비용이 Make보다 높음.

---

## 6. 어떤 상황에 적합한가

- **Make**: 조건 분기·다중 경로·에러 처리가 필요한 워크플로우, 비용을 절감해야 하는 개인·소규모 팀, 데이터 흐름을 직접 이해하며 만들고 싶은 경우에 적합.
- **Zapier**: "두 앱만 연결해서 세팅하고 잊고 싶은" 단순 선형 자동화, 팀이 이미 Zapier 생태계를 쓰고 있거나 희귀 앱 커넥터가 꼭 필요한 경우에 적합.

**결론**: 이번 과제처럼 조건 분기 + 멀티 Action이 필요한 워크플로우는 **무료 범위에서 모두 구현 가능한 Make가 더 적합**했다. Zapier는 동일 구조를 무료로 구현할 수 없어 Pro 체험이 필요했으며, 이 차이 자체가 두 도구의 핵심적인 차이점을 보여준다.

### 유료 기능 사용 사유 및 무료 대안
- Zapier **Paths(분기)** 는 Pro 전용 기능으로, 이번 구현에서는 Pro 14일 무료 체험을 활용했다.
- 완전 무료로 구현하려면 만족/불만족 각각 별도 Zap으로 분리하는 방식으로 우회 가능하나, 동일 구조 비교의 완성도가 낮아진다.
