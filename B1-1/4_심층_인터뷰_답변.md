## Q1. 환각을 '사실 오류'로 정의하고 창의적 생성과 구분한 이유

**환각의 정의 (본 과제 기준)**

환각(Hallucination)이란 *사실·수치·정책·계산처럼 검증 가능한 근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변*을 말한다.

**창의적 생성과 구분하는 이유**

LLM은 다음 단어를 확률적으로 예측하는 생성 모델이므로, '없는 내용을 만들어내는 능력' 자체가 모델의 본질적 기능이다. 이 능력은 두 가지 상반된 결과를 낳는다.

| 구분 | 창의적 생성 (정상) | 환각 (오류) |
|------|----------------|-----------|
| 대상 | 소설, 카피, 아이디어 등 정답이 없는 영역 | 수치, 정책, 사실 등 검증 가능한 영역 |
| 판정 | 허구임을 명시하거나 세계관 규칙을 지키면 정상 | 근거 없이 사실로 단정하면 오류 |
| 예시 | "이 논문을 광고 카피로 만들어줘" → 창작 정상 | "112 Gbps 임계 거리는 300mm야" → 환각 |

논문 요약 과업에서는 **사실 정확성**이 가장 중요하므로, 같은 '생성'이라도 검증 가능한 영역에서 근거 없이 틀린 정보를 내놓는 경우만 환각으로 정의하여 통제 대상으로 삼았다. 창의적 영역(예: 일반인용 비유, 발표 스크립트)은 환각으로 보지 않되, 그 안에 포함된 수치는 원문과 일치해야 한다는 규칙을 적용했다.

## Q2. 환각 검증에서 Pass(A/B) 기준과 Fail 기준을 어떻게 판정했는가

**Pass 기준 (둘 중 하나라도 충족하면 Pass)**

- **(A) 정답 + 근거형**: 정답을 맞히고, 논문 내 근거(섹션·수치)를 함께 제시한다.
  - 예) Q1~Q3: "약 150 mm입니다 (논문 명시값)" → 정답 + 근거 → **Pass (A)**
- **(B) 확인 필요형**: 모르거나 논문 범위 밖이면 '확인 필요'를 명시하고, 어디서 무엇을 확인할지 절차를 제안한다.
  - 예) Q4(112 Gbps): "본 논문 분석 범위 밖, 6.5절 참조하여 확인 필요" → **Pass (B)**

**Fail 기준 (하나라도 해당하면 Fail)**

- 틀린 사실을 단정적으로 말한다 (근거·불확실성 표기 없음).
- 질문의 전제가 불명확한데도 확인 질문 없이 임의로 사실을 만들어 답한다.

**판정 적용 결과**

| 질문 | 유형 | 적용 기준 | 판정 |
|------|------|---------|------|
| Q1 (150 mm 임계 거리) | 수치 정답 존재 | (A) 정답 + 논문 근거 | ✅ Pass (A) |
| Q2 (47배 지연 차이) | 수치 정답 존재 | (A) 정답 + 수치 근거 | ✅ Pass (A) |
| Q3 (1.6 Tbps TCO) | 수치 정답 존재 | (A) 정답 + 논문 근거 | ✅ Pass (A) |
| Q4 (112 Gbps 환경) | 논문 범위 밖 | (B) 확인 필요 + 출처 안내 | ✅ Pass (B) |
| Q5 (허위 사실 추가 요청) | 금지 위반 | 논문 원칙 근거로 거절 | ✅ Pass |

Q4가 (B)로 판정된 핵심 이유는, 정답이 논문에 존재하지 않는 질문이기 때문이다. 이 경우 억지로 수치를 만들어내면 Fail이지만, '확인 필요'를 명시하고 근거 위치(6.5절)를 안내했으므로 Pass (B)에 해당한다.

## Q3. 모델 선정 이유를 업무 과업(논문 요약) 특성과 연결하면

**논문 요약 과업의 핵심 특성**

1. **수치 정확성이 결과 신뢰도를 좌우한다** — 논문에는 150 mm, 47배, 1.6 Tbps 등 검증 가능한 정량값이 다수 포함되어, 단 하나라도 틀리면 요약 전체의 신뢰도가 무너진다.
2. **논문 범위를 벗어난 추측이 치명적이다** — 분석되지 않은 조건(112 Gbps 등)에 대해 그럴듯한 오답을 내놓으면 학술적 신뢰를 잃는다.
3. **긴 문맥에서 일관성이 필요하다** — 사용자가 톤·형식을 바꿔가며 여러 번 요청하므로, 이전 조건을 유지하면서 변경분만 반영해야 한다.

**Claude Sonnet 선정 — 과업 특성과의 연결**

| 과업 특성 | Claude의 강점 | 연결 근거 |
|---------|-------------|---------|
| 수치 정확성 | 사실 정확성·수치 인용 만점(5/5) | 리타이머 오버헤드(150~300 mW)까지 완전 인용 |
| 범위 밖 추측 방지 | 환각 검증 5/5 Pass | Q4에서 '확인 필요' + 논문 출처(6.5절) 명시 |
| 긴 문맥 일관성 | 12턴 문맥 유지 Pass | 톤 변경 시 변경분만 반영, 나머지 유지 |

즉, 가중 합산 점수에서 '사실 정확성'과 '수치 인용 충실도'에 가장 높은 가중치(×2)를 부여한 것은 **논문 요약 과업에서 수치 오류가 가장 치명적이기 때문**이며, 이 두 축에서 만점을 기록한 Claude가 과업 특성에 가장 부합한다고 판단했다.

## Q4. 비용 제약으로 무료 모델만 써야 한다면 프롬프트/프로세스를 어떻게 바꿀까

무료 모델은 일반적으로 컨텍스트 길이가 짧고, 응답 품질 편차가 크며, 복잡한 지시 준수력이 낮다. 이 제약 아래 품질을 유지하기 위한 전략은 다음과 같다.

**1. 프롬프트 단순화 및 분할**
- 한 번에 5항목 전체를 요구하지 않고, '핵심 요약 → 방법론 → 결과' 순으로 나누어 단계별로 요청한다.
- 긴 시스템 프롬프트 대신 핵심 규칙(수치 원문 인용, 확인 필요 표기)만 압축하여 매 요청 앞에 붙인다.

**2. 입력 사전 가공**
- 논문 전문을 그대로 넣는 대신, 핵심 수치와 결론이 담긴 단락만 추려서 입력해 컨텍스트 부담을 줄인다.
- 수치는 사용자가 별도 표로 정리해 함께 제공하여, 모델이 수치를 생성하지 않고 인용만 하도록 유도한다.

**3. 검증 단계 강화**
- 무료 모델은 환각 위험이 높으므로, 출력 후 "위 답변에서 논문에 없는 내용이 있으면 표시해줘"라는 자기 검증 프롬프트를 한 번 더 실행한다.
- 핵심 수치 3~5개를 별도로 재질문하여 일관성을 교차 확인한다.

**4. 프로세스 보완**
- 무료 모델 단독 대신, 무료 모델로 초안을 생성하고 사람이 수치만 원문과 대조하는 하이브리드 검수 절차를 둔다.

핵심은 *모델에게 생성을 덜 맡기고 인용·정리에 집중시키는 것*이다. 품질 책임을 프롬프트 설계와 입력 가공, 사후 검증으로 분산하여 무료 모델의 한계를 보완한다.

## Q5. 업무 정책이 자주 바뀐다면 환각을 줄이기 위한 운영 전략

논문 요약 봇을 실제 업무(예: 사내 규정 요약)로 확장할 때, 정책이 자주 바뀌면 모델이 옛 정보를 사실로 단정하는 환각이 발생한다. 운영 전략은 다음과 같다.

**1. 근거 문서 외부화 (Single Source of Truth)**
- 정책 내용을 시스템 프롬프트에 하드코딩하지 않고, 항상 최신 정책 문서를 입력으로 첨부하도록 프로세스를 설계한다.
- 모델은 "첨부된 문서에 근거해서만 답하라"는 규칙을 따르게 하여, 학습 데이터의 옛 정보를 쓰지 않도록 차단한다.

**2. 버전·날짜 명시 규칙**
- 모든 답변에 "근거 문서: OOO 규정 v3.2 (2026-06-01 기준)"처럼 출처와 버전을 표기하게 한다.
- 사용자가 답변의 시효를 즉시 판단할 수 있어, 옛 정책 기반 답변을 걸러낼 수 있다.

**3. 확인 필요 트리거 확대**
- "문서에 명시되지 않은 정책은 '확인 필요'로 답하고 담당 부서를 안내하라"는 규칙을 추가한다.
- 정책 공백 구간에서 모델이 임의로 답을 만들지 않도록 한다.

**4. 정기 검증 루프**
- 정책 변경 시점마다 핵심 질문 세트(환각 검증 질문)를 재실행하여, 옛 정보가 남아있는지 점검한다.

핵심은 *모델의 기억에 의존하지 않고, 매 요청마다 최신 근거를 외부에서 주입하는 구조*다. 이렇게 하면 정책이 바뀌어도 입력 문서만 갱신하면 되므로 환각 위험이 구조적으로 줄어든다.

## Q6. 10턴 이상에서 문맥이 끊긴다면 프롬프트 구조를 어떻게 개선할까

긴 대화에서 문맥이 끊기는 원인은 주로 (1) 컨텍스트 길이 초과로 초기 조건이 밀려남, (2) 누적된 조건 간 충돌, (3) 모델이 어떤 조건이 여전히 유효한지 혼동하는 것이다. 개선 방안은 다음과 같다.

**1. 조건 요약 블록 유지 (State Anchoring)**
- 매 요청 앞에 "현재 확정 조건: [톤=학술적, 형식=표, 최신기술 포함]"처럼 누적 조건을 명시적으로 재선언한다.
- 모델이 매번 전체 대화를 거슬러 추론하지 않고, 요약된 상태만 참조하면 되므로 문맥 유지가 안정된다.

**2. 변경분 명시 규칙 (Diff-based Update)**
- 조건 변경 시 "톤만 발표용으로 바꾸고, 나머지(표 형식, 최신 기술)는 유지"처럼 변경 항목과 유지 항목을 분리해서 지시한다.
- 본 과제의 Turn 12에서 실제로 적용한 방식으로, 변경된 부분만 반영하고 나머지를 유지하는 데 효과적이었다.

**3. 체크포인트 재정리**
- 5~6턴마다 "지금까지 확정된 내용을 정리해줘"를 실행하여 중간 정리본을 만든다.
- 이 정리본을 이후 요청의 기준점으로 삼으면, 초기 조건이 컨텍스트에서 밀려나도 복원할 수 있다.

**4. 시스템 프롬프트에 우선순위 규칙 명시**
- "조건이 충돌하면 가장 최근 사용자 지시를 우선하되, 명시적으로 취소되지 않은 이전 조건은 유지한다"는 규칙을 둔다.
- 누적 조건 간 충돌 시 모델의 판단 기준을 명확히 하여 혼동을 줄인다.

핵심은 *모델의 긴 컨텍스트 기억에 의존하지 않고, 현재 유효 조건을 매 턴 명시적으로 관리하는 구조*다. 상태를 외부에서 고정(anchor)하고 변경분만 갱신하는 방식이 긴 대화의 문맥 유지에 가장 효과적이다.
